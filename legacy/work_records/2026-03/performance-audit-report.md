# WABI 架构演进与并发性能研究报告
**日期**: 2026-03-28
**目标**: 解决 WABI 在高并发（200+ 用户）场景下的性能瓶颈与系统崩溃问题。

---

## 1. 核心问题与解决方案回顾

在将系统从单点测试推向 200 并发负载测试的过程中，我们暴露出了大量的架构瓶颈和事件循环阻塞问题。以下是基于第一性原理进行的改进记录：

### 1.1 节点饥饿与全局队列阻塞
- **遇到的问题**: 在早期的压测中，出现了大量闲聊 (`chitchat`) 节点超时的报错（如 `User 9 [chitchat] timed out`）。
- **原因分析**: 系统使用了一个全局的 AI Worker 池。当高耗时的节点（如上传图片的 `recognition` 或调用 Google Maps 的 `recommendation`）涌入时，它们会占满所有的 Worker，导致原本只需 0.5 秒即可返回的闲聊任务在队列中饿死。
- **解决方案**: **引入节点级语义背压 (Node-Level Semantic Backpressure)**。移除了对单点 Worker 的依赖，转而在 `semaphores.py` 中为不同的 Agent 提供独立的并发限流（例如：闲聊给 200，推荐给 30，识别给 10）。
- **是否成功**: 成功。实现了重负载与轻负载任务的解耦，防止了系统级饿死。
- **第一性原理**: 不同的任务消耗不同维度的资源（CPU、IO、内存、外部 API 额度）。用统一的队列对待异构任务，必然导致木桶效应。资源隔离是高并发的基石。

### 1.2 缓存击穿与持久化污染
- **遇到的问题**: Restaurant 推荐没有结果，且错误的状态被持久化到了 Redis 缓存中，导致同一个问题无论问多少次都在报错。
- **原因分析**: `GoogleMapsTool` 遇到 API Key 未加载或 HTTP 失败时，会返回空列表 `[]`，而旧逻辑不管三七二十一，把 `[]` 也缓存了 24 小时。
- **解决方案**: 修改 `google_maps.py`。失败的请求通过 `Exception` 抛出不写入缓存。对于正常的空结果，将 TTL 从 24 小时缩短至 1 小时。
- **是否成功**: 成功。缓存命中率在负载测试中达到 100%，且系统具备了自我恢复能力。
- **第一性原理**: 缓存的作用是阻挡重复计算，但错误的缓存等于固化了故障。系统必须对异常状态免疫。

### 1.3 静默失败 (Silent Failures) 与用户体验断层
- **遇到的问题**: 当后端某个 Agent 崩溃时（如请求超时），前端 Web 界面会一直转圈，直到 120 秒连接超时。
- **原因分析**: Worker Pool 捕获了异常，但没有将 `Exception` 转化为标准的结构化信息通过 Redis Pub/Sub 广播回对应的频道。
- **解决方案**: 在 `langgraph_server.py` 的拦截块中实现了**异常冒泡 (Error Bubble-up)** 机制，主动推回 `{"status": "error", "message": "..."}`。并在 `chat_manager.py` 中解析。
- **是否成功**: 成功。前端能秒级收到红色警告文本，提升了系统的可观测性。
- **第一性原理**: 错误是不可避免的，重点是 Fail-Fast（快速失败）。让调用方立刻知道失败原因并重试，好过让资源挂起 120 秒。

### 1.4 全链路事件循环阻塞 (Event-Loop Blocking)
- **遇到的问题**: 在 200 并发压测下，整个系统的延迟极其夸张。推荐节点平均延迟达 **55 秒**，视觉识别 **35 秒**，甚至原本极快的闲聊也飙升到了 **10 秒以上**。
- **原因分析**: 
  1. `ip_location.py` 中使用了**同步的 `requests.get()`**。在异步的 FastAPI 引擎中，同步的 I/O 会导致整个线程（事件循环）暂停。每一次 3-5 秒的 HTTP 请求都让 200 个并发请求排队等待。
  2. `fndds.py` 里面的 FAISS 向量检索和 `numpy` 运算在同步 `def` 中执行。这些 CPU 密集型任务每次会阻塞循环约 100ms。
  3. `llm_factory.py` 的单次调用设计导致每次生成新的 LLM Client 实例，这会在高并发下瞬间发起几百次 TLS 握手。
- **解决方案**:
  - 全面异步化：将 `ip_location.py` 替换为 `httpx.AsyncClient`，增加三级缓存（内存 -> Redis -> HTTP），并在服务器启动时强制预热。
  - CPU 任务卸载：将 `fndds_nutrition_search_tool` 改为 `async def`，并把 FAISS 查询包装进 `asyncio.to_thread`，放入线程池执行。
  - 单例模式复用：在 `llm_factory.py` 和 `google_maps.py` 中缓存对应的 Client 实例，复用底层 TCP 连接池。
- **是否成功**: 成功。打通了异步 IO 的任督二脉。
- **第一性原理**: Python asyncio 本质是单线程的。任何同步 IO 或重度 CPU 操作都会冻结整个世界（Block the World）。高并发异步系统的核心就是：永远不要阻塞事件循环。

### 1.5 内存泄漏与幽灵死锁
- **遇到的问题**: 审计发现 `langgraph_server.py` 为每一个短暂请求生成全局唯一的 `thread_id`，导致 `MemorySaver` 中的 checkpoint 字典无限膨胀；同时 Semaphores 存在多路复用下的幽灵事件循环绑定风险。
- **解决方案**: 脱落无状态请求的 Checkpointer；重构 `semaphores.py`，使用懒加载机制在运行期即时生成实例。

## 2. 下一步计划：攻克图像识别的并发难题

尽管当前的整体架构已经从早期的玩具级（Toy App）进化为能够承载几百 QPS 的异步流式系统，但我们在 **视觉识别 `recognition` 节点** 依然保留了一项巨大的并发隐患。

### 当前痛点
1. **网络带宽爆炸**: 每个食物识别请求都携带了完整的 Base64 图像（单张约 2~4MB）。当 200 个用户并发时，单个 Docker 容器的网络栈必须立刻承受约 ~600MB 的加密上传流量（发往大模型提供端）。
2. **算力瓶颈**: 即使 `fndds.py` 的检索被下放到了线程池，大量并发进来的图片特征提取与 RAG 工作依然极其消耗内存。
3. 当前只能靠 `recognition_semaphore = 10` 强行限制其并发。这并没有解决性能，只是防止了 OOM。

### 预期解决方案（下一步）
- 实施图像上云（S3/MinIO），向大模型传递 Presigned URL 而非原始 Base64，大幅减除内存拷贝和本地流量压力。
- 对视觉大模型建立专用的速率平滑队列（Rate-limiting queue），从大模型响应返回后才释放名额。
- 图片分辨率的前置降级优化。

本文件作为核心演进总结备存。所有的改动已发布在开发环境！

## 3. 性能优化成果 (200 并发负载测试对比)

经过一系列的异步化、连接池复用与线程池背压优化，最新的压测数据展示了跨越式的进步（**200 并发用户，1200 条消息轰炸**）：

### 📊 核心链路延迟对比
| 业务节点 | 优化前平均延迟 | 优化后平均延迟 | 提升幅度 | 解决的关键瓶颈 |
|---------|-------------|-------------|--------|--------------|
| **闲聊 (Chitchat)** | 10.32 秒 | **3.49 秒** | **约 66% 🚀** | 消除系统级事件循环阻塞，解绑大模型单例造成的数百次 TLS 连接握手 |
| **餐厅推荐 (Search)** | 55.34 秒 | **4.83 秒** | **约 91% 🚀** | 将同步 `requests` 替换为异步客户端，IP地址启动预热，Google Maps 建立 100% 命中缓存拦截网 |
| **图像识别 (Vision)** | 35.93 秒 | **22.06 秒** | **约 38% 🚀** | 引入 `asyncio.gather` 并发检索多个食物，FAISS 下放至 8-worker 独立线程池并受 Semaphore 严格背压限流 |

### 🛡️ 系统稳定性
- **单轮对话失败率**: 从早期过半数乃至雪崩的 Timeout 超时率，暴降至仅 **1.4%**（1200 次请求仅丢失17次）。
- **外部 API 保护**: 实施缓存后，高并发读写的 Google Maps 请求被 **100.0% 拦截**，瞬间挡住了 200 次并发外部 HTTP 请求开销。
- **总吞吐量**: 系统在 151.9 秒内从容完成了 1200 条夹杂着 2-4MB 图片处理与外部检索的复合请求集。
